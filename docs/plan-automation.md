# План автоматизации портала LKDS

## Фаза 1: Уведомления по дедлайнам ТЗ

### Что делаем
Флаги `overdue`, `deadline_soon`, `missing_deadline` уже считаются в `computeTzFlags()`.
Нужно: периодически проверять и отправлять уведомления.

### Реализация

**1.1. Cron-job / setInterval в AI-воркере**
- Каждое утро (09:00) воркер читает `tz.json`, считает флаги
- Формирует дайджест и отправляет в Telegram через существующий `TG_BOT_TOKEN`

**1.2. Формат утреннего дайджеста**
```
📋 Дайджест ТЗ — 2026-03-02

🔴 Просрочено: 3
  • TZ-ALIS-0002 «Отчёт продаж» — дедлайн анализа был 28.02
  • TZ-1C_CRM-0001 «Интеграция» — дедлайн разработки был 01.03
  • TZ-WMS-0003 «Инвентаризация» — дедлайн релиза был 25.02

⚠️ Скоро дедлайн (7 дней): 2
  • TZ-ALIS-0005 «Миграция» — анализ до 08.03
  • TZ-OTHER-0001 «Документация» — разработка до 07.03

🟣 Неполные дедлайны: 1
  • TZ-WMS-0001 «Склад v2» — в разработке, нет дедлайна разработки
```

**1.3. Мгновенные уведомления**
- При смене статуса ТЗ (PUT /api/tz/:id) проверять: если новый статус создаёт `overdue` — сразу уведомить
- При наступлении дедлайна (утренняя проверка) — уведомить ответственного

**1.4. Кому отправлять**
- Дайджест → `TG_ADMIN_IDS` (суперадмины)
- Персональные уведомления → по маппингу owner → telegram_id (нужно добавить поле в users.json)

### Файлы для изменений
- `scripts/ai-worker.js` — добавить cron-функцию утреннего дайджеста
- `server/app.js` — в PUT /api/tz/:id добавить проверку + уведомление при просрочке
- `data/users.json` — (опционально) поле `telegramId` для персональных уведомлений

## Фаза 2: Еженедельный ИИ-отчёт

### Что делаем
Раз в неделю (пн утро) AI-воркер читает данные и через Claude генерирует аналитический отчёт.

### Реализация

**2.1. Сбор данных**
Воркер собирает:
- Все ТЗ с флагами
- Историю изменений за неделю из `tz-history.json`
- Статистику: сколько ТЗ создано, завершено, застряло

**2.2. Промпт для Claude**
```
Ты аналитик проектного офиса. Вот данные по ТЗ за неделю:
[JSON с ТЗ и историей]

Сделай краткий отчёт:
1. Что продвинулось (какие ТЗ сменили статус)
2. Что застряло (в одном статусе > 14 дней)
3. Риски по дедлайнам
4. Рекомендации
Формат: Telegram HTML, коротко.
```

**2.3. Отправка**
- Результат Claude → Telegram дайджест суперадминам
- Сохранение в `data/ai-reports/` для истории

### Файлы для изменений
- `scripts/ai-worker.js` — добавить еженедельный cron
- Новая папка `data/ai-reports/`

## Фаза 3: Общая автоматизация (без ИИ)

### 3.1. Авто-эскалация застрявших ТЗ
- Если ТЗ в статусе `analysis` > 14 дней → автоматически поднять приоритет на 1 уровень
- Если ТЗ в `development` > 30 дней → уведомить + пометить в UI
- Пороги настраиваемые (конфиг или env)

### 3.2. SLA для ИТ-заявок
- Новая заявка без реакции > 4 часов → повторное уведомление
- Заявка `in_progress` > 3 дней → эскалация
- Автоматическое закрытие `done` заявок после оценки или через 7 дней

### 3.3. Напоминания о бронировании
- За 15 минут до встречи → Telegram-напоминание участнику
- Если комната забронирована но никто не пришёл (нет активности) > 30 мин → освободить

### 3.4. Дашборд руководителя
- Сводная страница: ТЗ + ИТ-заявки + CRM-тикеты в одном месте
- Графики: тренды создания/закрытия, среднее время в статусах
- Не требует ИИ — чистая агрегация данных

### 3.5. Шаблоны ТЗ
- Предустановленные шаблоны по типам: «Новый отчёт», «Интеграция», «Дефект»
- Автозаполнение полей: система, тип, типичный owner, чек-лист описания

### 3.6. Связи между ТЗ
- Поле `depends_on` / `blocks` — зависимости между ТЗ
- Визуализация: если блокирующее ТЗ застряло — подсветить зависимые
- Простой граф зависимостей

### 3.7. Автоматический код ИТ-заявки
- Сейчас ИТ-заявки без номера. Добавить: `IT-2026-0001`
- Удобнее ссылаться в переписке

### 3.8. Повторяющиеся задачи
- Шаблон «создавать ТЗ каждый месяц»: ежемесячные отчёты, регулярные проверки
- Cron в воркере создаёт ТЗ по шаблону
